# Journal Implementation Plan

## Summary

Текущий приоритет проекта: как можно раньше обеспечить надежный бэкап пользовательских данных, а уже после этого переходить к глубоким переделкам интерфейса и доменной модели. Поэтому первая очередь состоит из архитектурной перестройки под FSD 2.1, добавления страницы настроек и реализации импорта/экспорта текущих данных.

Ключевые зафиксированные решения:

- Приложение остается mobile-first offline PWA.
- Данные продолжают храниться в IndexedDB.
- Архитектурная структура: `app`, `pages`, `shared`.
- `shared` разбит на purpose-based сегменты:
  - `ui` для переиспользуемого UI;
  - `routes` для маршрутов;
  - `types` для сущностей и базовых доменных типов;
  - `lib/*` для узко сфокусированных внутренних библиотек с задокументированными границами.
- Основные page-slices: `calories`, `collections`, `settings`.
- UI-библиотека для следующего большого этапа: `PrimeVue`.
- Импорт объединяет данные с текущими, а не заменяет их целиком.
- Полная замена данных делается только через отдельную очистку.
- Стратегия бэкапа:
  - сейчас: versioned JSON для текущих данных;
  - позже, после добавления фото: единый архивный файл приложения.

## Phase 1: Safe Foundation

Цель этапа: перестроить проект и получить бэкап без заметной ломки текущего поведения.

Задачи:

- Перенести код на структуру FSD 2.1.
- Разложить страницы по `pages/calories`, `pages/collections`, `pages/settings`.
- Перенести общие маршруты, сущности, UI и внутренние библиотеки в `shared`.
- Сохранить текущие сценарии `collections` и текущую модель калорий без содержательной переработки.
- Добавить страницу `settings`.
- Реализовать экспорт текущих данных в versioned JSON.
- Реализовать импорт текущих данных с merge по первичному ключу.
- Реализовать отдельную кнопку полной очистки данных.
- Покрыть `export/import/clear` и миграции минимальными автотестами.

Критерии готовности:

- Приложение собирается и запускается.
- Все старые экраны продолжают работать.
- Из `settings` можно выгрузить backup, очистить локальные данные и восстановить их из файла.
- `typecheck`, `lint`, `test` и `build` проходят.

## Phase 2: Collections After Backup

Цель этапа: после появления безопасного бэкапа спокойно переработать более стабильную часть приложения.

Задачи:

- Довести `collections` до новой архитектуры без наследия старой структуры.
- Упростить и вычистить связанную логику создания/редактирования коллекций и item.
- Начать замену `daisyUI` на `PrimeVue`, но делать это постепенно, сохраняя работоспособность приложения на каждом подэтапе.

Критерии готовности:

- Сценарии `collections` работают поверх новой структуры без временных костылей.
- Общие компоненты и навигация готовы к дальнейшей миграции UI.

## Phase 3: Photo Field

Цель этапа: добавить новый тип поля `photo` и подготовить backup-формат к бинарным данным.

Продуктовые решения:

- Поле `photo` поддерживает несколько изображений.
- Все фото хранятся в отдельном object store IndexedDB.
- Каждое фото имеет собственный `id`.
- Фото связано как минимум с `collectionId`.
- В `item` хранятся только ссылки на фото, то есть массив `photoId[]`.

Задачи:

- Добавить новый `CollectionFieldKind = 'photo'`.
- Добавить отдельное хранилище `photos` в IndexedDB.
- Сделать UI выбора фото, предпросмотра и удаления.
- По возможности использовать мобильную съемку через file input с `capture`.
- Расширить import/export под фотографии.
- Перевести backup-формат с JSON на единый архивный файл приложения:
  - `manifest`
  - бинарные файлы фото

Критерии готовности:

- В item можно прикреплять несколько фото.
- Фото корректно восстанавливаются после экспорта/импорта.
- Пользователь по-прежнему работает с одним backup-файлом.

## Phase 4: Calories Redesign

Цель этапа: полностью заменить текущую сырую реализацию учета калорий.

Продуктовые решения:

- Новый экран должен быть похож на калькулятор.
- Сверху находится выбор дня.
- Ниже находится прокручиваемый список выражений за выбранный день.
- Снизу находится сетка `4x5` как у калькулятора.
- Каждый день хранится через отдельные поля `year`, `month`, `day`.
- Старую модель калорий не нужно сохранять содержательно.

Задачи:

- Ввести новую модель калорийного дня.
- Хранить в дне список выражений.
- Считать итоговую сумму как сумму значений всех выражений дня.
- Реализовать новый mobile-first UX для быстрого ввода.

Критерии готовности:

- Калорийный экран работает независимо от timezone-сдвигов.
- Пользователь быстро вводит и редактирует выражения с телефона.
- Итог по дню всегда вычисляется из списка выражений.

## Test Strategy

Автотесты нужны только там, где они реально защищают данные и миграции.

Покрывать автоматически:

- экспорт текущих данных;
- импорт поверх существующих данных;
- merge по первичному ключу;
- полную очистку данных;
- миграции IndexedDB;
- позже: ссылочную целостность `item -> photoIds` и экспорт/импорт фото.

Не покрывать на текущем этапе:

- старую калорийную логику, так как она будет почти полностью заменена.

Ручная проверка обязательна для:

- offline/PWA установки;
- основных мобильных сценариев;
- импорта/экспорта через реальный UI;
- фото-потока после его добавления;
- нового calorie UX после редизайна.
